== Building Block View

=== Level 1 — Whitebox ultrathink.call

The static app decomposes into seven building blocks. The external systems are
the same set as Chapter 3. Since EPIC 1 (issue #7, done) the blocks live in
separate ES modules under `src/app/` — `codec.js`, `peer.js`, `signaling.js`,
`mesh.js`, `main.js` — loaded natively without a build; the source-location
column names each module.

=== Building blocks

|===
| Block | Responsibility | Interface | Source location
| BB-1 UI Shell | A guided wizard (#46): one-step *Call eröffnen* / *Beitreten* (media + invite/answer together), progressive disclosure (in-call controls appear with media, none greyed-out), an explicit *keep-page-open* note + `beforeunload` guard (the pending connection lives only in memory, ADR-8), and the answer rendered as a copy/paste token — never an openable link — with an `#answer=` load guard. The invite is a compact link (#44). | DOM events; `status()`; `setLink`/`linkUrl`; `arm/disarmUnloadGuard`; `showStep()` over the mutually-exclusive phases (#50). | `index.html` (phase sections); `src/app/main.js` (event wiring)
| BB-2 Media Manager | Capture camera/mic, enumerate and switch devices (per kind), screen share (feature-detected), restore camera, and local self-mute (#47, `track.enabled` for mic/cam, re-applied after a device switch). | `getUserMedia`, `enumerateDevices`, `replaceTrack`, `track.enabled`. | `src/app/main.js`: `populateDevices`, `switchTrack`, `restoreCamera`, `applyMic`/`applyCam` + the in-call control handlers
| BB-3 Peer Connection | Own the `RTCPeerConnection`: tracks, offer/answer, ICE (non-trickle, timeout-bounded), and a `ctrl` data channel used by the mesh. | `PeerSession` class; `waitForIce`; `sendCtrl`. | `src/app/peer.js`
| BB-4 Signaling Transport (port) | Produce and consume capability tokens; the manual adapter distributes them over three channels — native share (Web Share API, #42), `mailto:`, and clipboard — chosen by capability, no broker (ADR-8). | `buildCapabilityUrl`, `parseIncoming`, `mailtoLink`, `shareData`, `canNativeShare`. | `src/app/signaling.js`; `src/app/main.js` (DOM glue) — see ADR-5
| BB-5 Token Codec | Compress and encode the SDP payload to a URL-safe token and back. | `encodePayload`, `decodePayload`, `extractToken`. | `src/app/codec.js` (pure, unit-tested)
| BB-6 Validation | Enforce decode-time rules VR-1..6,8,9 and the post-decode guard GD-1. | validation inside `decodePayload`; guard in handlers. | `src/app/codec.js` (`decodePayload`); `src/app/main.js` (`if (mesh)` guards)
| BB-7 Mesh Orchestrator | Manage the peers (`Map` of PeerSessions); host bootstraps each guest and relays guest↔guest signaling (ADR-8). Control protocol over `ctrl`: `welcome` (host→guest: your id + peers to connect to), `signal` (offer/answer routed by `to`), the host-only moderation verbs `kick`/`mute` (M-8), and `chat` (#48 — direct P2P broadcast, no relay, attributed to the channel it arrived on, text-only render BR-8). Relayed SDP is validated (DTLS fingerprint, like VR-9); the host stamps the authoritative `from`; peer count is capped; guests honour `kick`/`mute` only from the host session. | `MeshSession` (`createBootstrapOffer`, `acceptBootstrapAnswer`, `joinWithOffer`, `kick`, `setMuted`, `sendChat`, `_handleCtrl`). | `src/app/mesh.js`
| BB-8 Adaptive Media Controller | Decide the local video encoding tier under load (#27, S-4, QS-3): given a periodic "any sender CPU/bandwidth limited?" sample, step the tier down after sustained pressure and back up after sustained calm. Pure decision logic (IOSP Operation); the floor tier holds ≥ 10 fps (SM-3). main.js samples `getStats` and applies the verdict via `setParameters` to every peer's video sender (IOSP Integration). | `AdaptController` (`sample`, `tier`); `PeerSession.videoLimitation`, `PeerSession.applyVideoParams`. | `src/app/adapt.js`; `src/app/peer.js`; `src/app/main.js` (sampling loop)
| BB-9 Video Processing (port) | Transform the outgoing video before it is sent — the third adapter slot beside signaling and ICE (S-5, ADR-5, G-6). The one adapter is native background blur (#25, ADR-9): delegate to the platform's `backgroundBlur` MediaStreamTrack capability, no model fetched, so CSP and sovereignty are untouched. Offered only where supported; because `backgroundBlur` is a best-effort *advanced* constraint, enabling is confirmed by a `getSettings()` read-back and rejected otherwise — so a blur not actually in effect is never reported as on (fail-closed). A future vendored-model adapter (#26) implements the same port. | `supportsBlur`, `NativeBlurProcessor` (`supported`, `setEnabled`). | `src/app/blur.js`; `src/app/main.js` (toggle + device-switch re-apply)
|===

=== Level 2 — Mesh and the signaling seam

BB-4 is the architecture's main seam (QG-4): a transport-agnostic token
contract with one adapter (the manual capability-URL + `mailto:`), no broker
(ADR-8). BB-7 builds the mesh on top: host↔guest links are bootstrapped through
BB-4, then the host relays guest↔guest SDP over the BB-3 `ctrl` channels until a
full mesh forms. The host relays *signaling only* — media stays direct P2P
(DTLS-SRTP). Adding the ~20 training topology (ADR-4) would extend BB-7 without
touching BB-3 or BB-5.
