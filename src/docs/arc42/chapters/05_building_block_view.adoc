== Building Block View

=== Level 1 — Whitebox ultrathink.call

The static app decomposes into seven building blocks. The external systems are
the same set as Chapter 3. Since EPIC 1 (issue #7, done) the blocks live in
separate ES modules under `src/app/` — `codec.js`, `peer.js`, `signaling.js`,
`mesh.js`, `main.js` — loaded natively without a build; the source-location
column names each module.

=== Building blocks

[cols="1,2,2,2",options="header"]
|===
| Block | Responsibility | Interface | Source location
| BB-1 UI Shell | Present controls, video tiles, status; wire DOM events to the other blocks. | DOM events; `status()` text. | `index.html` (markup); `src/app/main.js` (event wiring)
| BB-2 Media Manager | Capture camera/mic, enumerate and switch devices (per kind), screen share (feature-detected), restore camera. | `getUserMedia`, `enumerateDevices`, `replaceTrack`. | `src/app/main.js`: `populateDevices`, `switchTrack`, `restoreCamera` + `#startCam`/`#shareScreen` handlers
| BB-3 Peer Connection | Own the `RTCPeerConnection`: tracks, offer/answer, ICE (non-trickle, timeout-bounded), and a `ctrl` data channel used by the mesh. | `PeerSession` class; `waitForIce`; `sendCtrl`. | `src/app/peer.js`
| BB-4 Signaling Transport (port) | Produce and consume capability tokens; the one adapter is capability-URL + `mailto:` + clipboard (no broker, ADR-8). | `buildCapabilityUrl`, `parseIncoming`, `mailtoLink`, `inviteTokenInHash`. | `src/app/signaling.js`; `src/app/main.js` (DOM glue) — see ADR-5
| BB-5 Token Codec | Compress and encode the SDP payload to a URL-safe token and back. | `encodePayload`, `decodePayload`, `extractToken`. | `src/app/codec.js` (pure, unit-tested)
| BB-6 Validation | Enforce decode-time rules VR-1..6,8,9 and the post-decode guard GD-1. | validation inside `decodePayload`; guard in handlers. | `src/app/codec.js` (`decodePayload`); `src/app/main.js` (`if (mesh)` guards)
| BB-7 Mesh Orchestrator | Manage the peers (`Map` of PeerSessions); host bootstraps each guest and relays guest↔guest signaling (ADR-8). Control protocol over `ctrl`: `welcome` (host→guest: your id + peers to connect to) and `signal` (offer/answer routed by `to`). Relayed SDP is validated (DTLS fingerprint, like VR-9); the host stamps the authoritative `from`; peer count is capped. | `MeshSession` (`createBootstrapOffer`, `acceptBootstrapAnswer`, `joinWithOffer`, `_handleCtrl`). | `src/app/mesh.js`
|===

=== Level 2 — Mesh and the signaling seam

BB-4 is the architecture's main seam (QG-4): a transport-agnostic token
contract with one adapter (the manual capability-URL + `mailto:`), no broker
(ADR-8). BB-7 builds the mesh on top: host↔guest links are bootstrapped through
BB-4, then the host relays guest↔guest SDP over the BB-3 `ctrl` channels until a
full mesh forms. The host relays *signaling only* — media stays direct P2P
(DTLS-SRTP). Adding the ~20 training topology (ADR-4) would extend BB-7 without
touching BB-3 or BB-5.
