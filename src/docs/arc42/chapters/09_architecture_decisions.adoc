== Architecture Decisions

ADRs follow Nygard with a 3-point Pugh matrix (`-1`/`0`/`+1` against a datum
option). Consequences name the Chapter 11 risks each decision creates.

=== ADR-1 Server-less peer-to-peer, no own infrastructure

*Status*: Accepted. *Context*: independence from US providers without running or
paying for infrastructure (G-1, G-2). *Decision*: a static page plus browser
WebRTC; peers connect directly, no media server.

[cols="2,1,1,1",options="header"]
|===
| Criterion | SaaS (Zoom) — datum | Self-hosted SFU | Server-less P2P
| Sovereignty (G-1) | 0 | +1 | +1
| Zero-ops (G-2) | 0 | -1 | +1
| Scales past ~6 peers | 0 | +1 | -1
| Cost | 0 | -1 | +1
| *Sum* | *0* | *0* | *+2*
|===

*Consequences*: chosen for sovereignty + zero-ops; creates *R-1* (mesh cannot
reach ~20 — the central tension) and inherits *R-6* (IP exposure in P2P).

=== ADR-2 Capability-URL signaling (SDP-in-link + `mailto:`)

*Status*: Accepted. *Context*: signaling without a backend or login (G-2, G-3).
*Decision*: carry the SDP in the URL fragment; the token is the bearer secret;
distribute via the host's own `mailto:`. The automatic broker is a later adapter
(R2), not a reason to add infrastructure now.

[cols="2,1,1,1",options="header"]
|===
| Criterion | Email magic-link service — datum | Public broker (Trystero) | Capability-URL + mailto
| Zero backend (G-2) | 0 | +1 | +1
| No login (G-3) | 0 | 0 | +1
| Usability at 2 peers | 0 | +1 | 0
| Sovereignty (G-1) | 0 | -1 | +1
| *Sum* | *0* | *+1* | *+3*
|===

*Consequences*: chosen for full sovereignty and zero backend; creates *R-3*
(token leak/replay — no revocation) and is verbose at scale. R2 keeps this for
host↔guest bootstrap and adds host-relayed guest↔guest signaling (ADR-8) rather
than a broker. DTLS-fingerprint exchange over the capability channel partially
mitigates *R-7*.

=== ADR-3 Sovereign (non-US) STUN; no Google

*Status*: Accepted. *Context*: G-1 forbids any US-jurisdiction runtime component,
"not even STUN". *Decision*: use a non-US STUN (Nextcloud, DE) as the interim
endpoint; the vetted set with fallbacks is D-9/R2.

[cols="2,1,1,1",options="header"]
|===
| Criterion | Google STUN — datum | No STUN (host candidates only) | Sovereign STUN (DE)
| Sovereignty (G-1) | 0 | +1 | +1
| Works across NAT | 0 | -1 | 0
| Reliability | 0 | -1 | -1
| *Sum* | *0* | *-1* | *0*
|===

The chosen option ties the Google datum on raw count: a single interim DE
endpoint is genuinely less reliable than Google's anycast (the R-2 cost, now
visible in the matrix). The tie is broken by *QG-1 (sovereignty)*, the
top-ranked quality goal (Chapter 1.2).

*Consequences*: removes the G-1 violation in the media path. D-9 is now done —
a sovereign (DE) STUN set with fallbacks (Nextcloud, sipgate, 1&1) replaces the
single interim endpoint, reducing *R-2* (a unit test asserts no US host, SM-2).
Uptime monitoring of the chosen endpoints stays operational.

=== ADR-4 Topology for the ~20-participant training case

*Status*: Accepted (inferred). *Context*: a full mesh uploads `N-1` streams per
peer, so at ~20 every home uplink dies. The textbook fix is an SFU — but an SFU
is infrastructure to operate and a non-US-relay trust question, which G-2 (no own
infrastructure) and G-1 (sovereignty) rule out. The remaining infra-free lever is
*who sends video*. *Decision*: **asymmetric broadcast** — in training mode only
the presenter(s) send video; participants are audio-only by default and may be
promoted to send video one at a time (raise-hand). This keeps the zero-infra P2P
posture. It does *not* make the mesh scale arbitrarily: the presenter still
uploads one video copy per participant, so the realistic ceiling is the
presenter's uplink — accepted as a **cap** (PRD option c folded in), not a promise
of 20-with-everyone-on-video. True 20-with-video for all needs the deferred
relay/SFU infra decision (ADR-bound, ties to the TURN question #4).

[cols="2,1,1,1",options="header"]
|===
| Criterion | Full mesh — datum | Asymmetric broadcast (chosen) | Self-hosted EU SFU
| Sovereignty (G-1) | 0 | 0 | 0
| Zero-ops (G-2) | 0 | 0 | -1
| Reaches ~20 (audio + 1 presenter video) | 0 | +1 | +1
| UX symmetry (everyone on video) | 0 | -1 | +1
| No new trust/infra | 0 | +1 | -1
| *Sum* | *0* | *+1* | *0*
|===

Asymmetric broadcast wins on the sum and, decisively, on the two top-ranked goals
(QG-1 sovereignty, QG-2 zero-ops) where the SFU pays `-1`. The cost it accepts is
UX symmetry — in a training, participants are not all on camera at once. For the
owner's actual use (one trainer, ~20 listeners) that matches the real format.

*Consequences*: chosen for sovereignty + zero-ops at the price of symmetric video.
*Reduces R-1* (the topology tension now has a decided, infra-free answer) but does
not close it: the presenter-uplink cap is a new accepted limitation, *R-15*. The
implementation (issue #31, MoSCoW *Could*) adds presenter/participant roles and
per-role send policy to BB-7; until built, R5 stays vision. A later relay/SFU
decision (#4) could lift R-15 if the owner ever accepts that infrastructure.

=== ADR-5 Ports and adapters for external dependencies

*Status*: Accepted. *Context*: avoid new lock-in; allow swapping signaling, ICE,
and video processing (G-6). *Decision*: each external dependency sits behind a
port; adapters form an anti-corruption layer.

[cols="2,1,1",options="header"]
|===
| Criterion | Inline in handlers — datum | Ports + adapters
| Swappable signaling (G-6) | 0 | +1
| Unit-testable codec | 0 | +1
| Slice speed at R1 | 0 | -1
| *Sum* | *0* | *+1*
|===

*Consequences*: chosen for modifiability and testability. The seam was pulled
forward and realized in EPIC 1 (issue #7 done, see ADR-7) rather than deferred to
R2 — the port + adapters now live in `src/app/` and TD-1 is resolved. Creates no
new tracked risk.

=== ADR-6 Static host: GitHub Pages as interim, not sovereign

*Status*: Accepted (inferred, interim). *Context*: the static host delivers the
executable client code and is therefore a runtime endpoint that G-1/TC-2 require
to be non-US. The project currently deploys to *GitHub Pages* (Microsoft, US),
which contradicts that goal — surfaced by the ATAM review. *Decision*: accept
GitHub Pages as an explicit interim while the app is a tracer, and treat the
move to a sovereign EU static host as required before any sovereignty claim
(SM-2) is presented as met. The product owner chose *Codeberg Pages (DE)* as the
target; the migration is expected to be trivial and is carried as technical debt
*TD-6*.

[cols="2,1,1,1",options="header"]
|===
| Criterion | GitHub Pages — datum | EU sovereign Pages (Codeberg / GitLab-EU) | Self-hosted EU
| Sovereignty (G-1) | 0 | +1 | +1
| Zero-ops (G-2) | 0 | 0 | -1
| Migration effort | 0 | -1 | -1
| *Sum* | *0* | *0* | *-1*
|===

The Pugh tie (datum vs. EU Pages) is again broken by QG-1: EU Pages wins on the
top-ranked goal at no zero-ops cost. The cell that needs owner input — the exact
host — is left to the decision in issue (host migration).

*Consequences*: until migrated, the deployed app violates G-1; recorded as
*R-11* (high) and technical debt *TD-6*, tracked in issue #10. Development and
source on GitHub are unaffected (not a runtime endpoint); only the *delivery*
host is in scope.

=== ADR-7 ES modules + dev-only test tooling (no app build)

*Status*: Accepted (inferred — EPIC 1). *Context*: realizing the ports concept
(ADR-5) and the test concept (8.3) needs modular, importable code, but the app
was a single inline-script file (good for CSP, bad for testing). *Decision*:
split the JavaScript into native ES modules under `src/app/` loaded with
`<script type="module">` — no bundler, no app build (TC-1 holds) — and add
*vitest* (unit) and *Playwright* (E2E) as dev-only dependencies that never ship.

[cols="2,1,1",options="header"]
|===
| Criterion | Single inline file — datum | ES modules + dev test tooling
| Unit-testable codec | 0 | +1
| Swappable adapters (G-6) | 0 | +1
| Stays build-less, no shipped deps | 0 | 0
| Inline-script attack surface | 0 | +1
| *Sum* | *0* | *+3*
|===

*Consequences*: the inline `<script>` is gone, so CSP tightens to
`script-src 'self'` — this closes the T-010 gap M-2 previously leaned on M-3 for.
Dev dependencies are added but never served (the app is still static, no build).
Creates no new tracked risk.

=== ADR-8 Mesh without a broker — host-relayed signaling

*Status*: Accepted (owner decision, EPIC 2). *Context*: the 4-peer mesh (R2)
needs automatic guest↔guest signaling. An external broker (even a sovereign one)
adds a runtime dependency and a non-US-relay trust question; the owner chose to
keep full sovereignty and the zero-dependency posture. *Decision*: no broker.
Host↔guest links are bootstrapped manually (capability-URL, one invite per
guest); the host then *relays the tiny SDP control messages* between guest pairs
over the `ctrl` data channels until a full mesh forms. The host relays signaling
only — media stays direct P2P (DTLS-SRTP), never through the host.

[cols="2,1,1,1",options="header"]
|===
| Criterion | Vendored broker (Trystero) — datum | Self-built relay | Host-relayed (no broker)
| Sovereignty (G-1) | 0 | +1 | +1
| Zero dependencies / no build | 0 | 0 | +1
| Host UX at 4 | 0 | +1 | -1
| Scales toward ~20 | 0 | +1 | -1
| *Sum* | *0* | *+3* | *0*
|===

The self-built-relay option scores higher on paper but still needs an external
relay endpoint; the owner weighted *no external relay at all* above UX and
scale, breaking the tie toward host-relayed (QG-1 top-ranked).

*Consequences*: full sovereignty and zero dependencies; the host does one
capability-URL exchange per guest (clunkier UX, accepted), and is a signaling
single-point-of-failure *during setup* (not for established media). Because the
host relays guest↔guest SDP, a *malicious host* could rewrite the fingerprints
and MITM guest↔guest media (R-7) — *accepted residual*, since the host is the
trusted meeting organizer. Against a malicious *guest*, the relayed SDP is still
validated (DTLS fingerprint, VR-9 on the mesh) and the host stamps the
authoritative sender id, so a guest cannot downgrade encryption or impersonate a
peer. Practically caps at a small mesh — the ~20 training case remains
R-1/ADR-4. Realized in `src/app/mesh.js`.

=== ADR-9 Background blur via the platform, not a vendored model

*Status*: Accepted (owner decision, R3). *Context*: R3 wants client-side
background blur (G-5, SM-4) — the privacy feature that should beat Jitsi's weak
virtual background. Real segmentation needs a model. A model has to come from
somewhere, and every "somewhere" collides with a core principle: an external CDN
breaks the strict CSP (`script-src 'self'`, ADR-7) and sovereignty (G-1); a
*vendored* model adds multi-MB binaries to the repo and reopens the supply-chain
threat (T-003 / R-10), forcing the #26 gate (SRI + Trusted Types) before it can
ship. *Decision*: delegate to the *platform's own* background blur — the
MediaStreamTrack `backgroundBlur` capability (ChromeOS, Windows Studio Effects) —
applied through the video-processing port (BB-9). No model is fetched, so CSP and
sovereignty hold and #26 is *not* triggered. Where the platform cannot blur, the
toggle is unavailable and we never report a blur we cannot deliver (fail-closed).

[cols="2,1,1,1",options="header"]
|===
| Criterion | Vendored WASM model — datum | Native platform blur | External CDN model
| Sovereignty (G-1) | 0 | 0 | -1
| Strict CSP unchanged (ADR-7) | 0 | +1 | -1
| No new supply-chain risk (T-003) | 0 | +1 | -1
| Blur quality / coverage | 0 | -1 | 0
| Works on every browser | 0 | -1 | 0
| *Sum* | *0* | *0* | *-3*
|===

The native option ties the vendored datum: it trades blur quality and universal
availability for an unchanged CSP and no new supply-chain risk. The owner broke
the tie toward *native* — keep the principles intact now, accept that blur is
only offered where the platform delivers it.

*Consequences*: blur ships with no model, no CSP change, no new dependency; on
platforms without the capability the feature is simply absent (a coverage gap,
*R-14*, not a security risk). The vendored-model path stays open as a future
video-processing adapter behind the same port; if it is ever taken, #26
(SRI/Trusted Types, T-003/R-10) becomes a hard prerequisite again. Fail-closed
(arc42 8.5) is adapted for the no-model case: we never present an unblurred frame
as blurred — on any failure the toggle reverts and warns. Realized in
`src/app/blur.js`.
